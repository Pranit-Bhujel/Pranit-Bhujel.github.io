Variables¶

Variables are actually memory references. When an object is stored in the memory, it may occupy more space or less depending on the type of the variable. Python Memory Manager handles the object creation in the heap memory.

We can always find the memory address that the variable is pointing at by using id() function. The hex code of the memory address can be found using hex(id(var_name))

Reference Counting¶

How many pointers are pointing to or referencing a particular memory address? This is something called as reference count. For a particular memory address, the total number of variables pointing to it will be stored as counts.

There are two ways to find out the reference count for a particular variable/memory address. Also to remember that a varibale is just a pointer to the memory address.

In [17]:
'''This method adds one extra value to the reference count'''
import sys

var = '10'

sys.getrefcount(var)
Out[17]:
5
In [8]:
'''Adding a few more references'''
a = var
b = var
In [10]:
'''This method gives the exact reference count, but actually takes the address directly than the variable name'''
import ctypes

ctypes.c_long.from_address(id(var)).value
Out[10]:
6

So, what's the main role played by reference counting? The answer is to keep track of the active objects, so as an when the reference falls down to 0, the Python Memory Manager sweeps the object to free the space!

Garbage Collection¶

Circular Reference: When there's a circular reference that keeps the reference count greater than 0 and so the reference count mechanism is not always suitable to free the unused objects. In this case the garbage collector does it work.

It is not recommended to use Python V3.4 and before, as the python interpreter gets confused over this criteria; If __del()__ destructor is defined in the class of the object that is stuck in the circular reference, the gc would get confused as in what sequence should it follow to destroy those objects and hence would not be able to destroy!

This causes the object to be marked as uncollectable and the objects in the circular reference are not cleaned up - a term called Memory Leak.

It was however solved in the later python versions. Now, garbage collector runs periodically to sweep extra objects which are not necessary for the program and hence consumes a good chunk of computational power.

Also, the garbage collector can be disabled using the gc module in python. However, it is not recommended as the memory allocation is pretty unpredictable, so it should not be disabled unless you're absolutely certain of what you're doing! By default the garbage collector is turned on.

Statically Typed vs Dynamically Typed¶

If we look at some programming languages like C, C++ and Java, we'll notice that the variable declaration has three parts:

Statically Typed:

int a = 10;

Now, if we look into Python we just have to write

Dynamically Typed:

x = 10

So the only difference is that, in statically typed languages, we explicitly define the variable to be of a certain type. Hence if we try to allocate a value other than that particular data type, it will throw an error.

However in dynamically typed language, the variable is merely a reference to the memory location and hence we can change it to point to any other memory location and it won't throw an error!

Variable Re-assignment¶

When we reassign a value to a variable, then in python we'll see that the variable will now reference to a different object in the heap memory rather than updating the new value in the previous memory location.

This involves a concept called object mutability and immutability which we'll see further down in the article. Let's see how variable reassignment works.

In [11]:
a = 10

hex(id(a))
Out[11]:
'0x1d668096a50'
In [14]:
'''when we update a, it points to a different object in the memory now.'''
a = a + 1

hex(id(a))
Out[14]:
'0x1d668096b10'
In [12]:
a = 15

hex(id(a))
Out[12]:
'0x1d668096af0'
In [13]:
b = 15

hex(id(b))
Out[13]:
'0x1d668096af0'

In the second example, we can see that both a and b point to the same memory location, despite having different names.

Object Mutability & Immutability¶

Changing the data inside the object is called as modifying the internal state of the object.

An object whose internal state can be changed is called as Mutable Object whereas an object whose internal state cannot be changed is called as an Immutable Object.

Immutable Mutable
Numbers(int, float, booleans Lists
Strings Sets
Tuples Dictionaries
Frozen Sets User-Defined Classes
User Defined Classes

For an immutable collection like tuple, it should be noted that the elements may be mutable or immutable.

So, the immutable object having mutable elements can indeed be modified!

Function Arguments and Mutability¶

Immutable Objects are safe from unintended side effects.

The big picture here is that, immutable objects do not change unless we have mutable elements in that object, say for example list in a tuple. So, we have to be careful when we say immutable objects cannot be changed.

Shared References and Mutability¶

Python Memory Manager is smart, in the sense that when it creates a shared reference to a memory address, it shares the same location with the immutable objects but creates different memory location for the mutable objects.

The good reason for doing that is, say we define a=[1, 2, 3] and we the write b=a, and now if we change b by adding or removing any element then it'll affect a as well.

So when mutable objects are created, the python memory manager creates them at different memory location even if they have the same content.

In [1]:
a = [1, 2, 3]
b = [1, 2, 3]
print(id(a))
print(id(b))

Equality¶

Measuring equality can be done using two different approaches.

  1. Comparing the object address.
  2. Comparing the internal state of the object.

The first one is achieved using the operators is and is not, called the identity operators. To measure the internal state of the object we use the comparison operators == and !=.

Magic of None: It's not that the PMM will create different None objects but will point to same None object everytime we create a None type variable.

Everything is an object!¶

In Python everything is an object. So when we declare an integer type varible or a function or anything, then an instance of that particular class is created. That is why it is an object.

Now, what are the advantages of that being an object? So, when an object is created, the function name or the variable name is just a reference to the object. So, it allows us to achieve the following.

  • Assign a function to a variable.

  • Have function parameters as function.

  • Return a function through a function.

When you pass the function reference to a variable in python, the variable now works the same as the function.

When we use function name without the parenthesis, we are just referring to the object in the memory, but when we use the parenthesis, we're actually invoking a function
