Python Naming Conventions, The Mystry of Underscores

"Underscores play a vital role in python as it allows interpreters to identify the type of variable or a method."

The General Notion:¶

Variables/Identifiers are case sensitive and can have letters, _, and digits. It can begin with letters and _, but not with a digit.

Single Underscore in the Front¶

This is a convention to indicate that the identifier is to be used for internal use or private objects. This kind of objects will not get imported by an import statement such as from module import *.

However there are other ways of accessing it.

. . .¶

Double Underscore in the Front¶

This convention is primarily used for something called as Name Mangling. Name Mangling is a process in which an identifier with a double underscore in the front __name is replaced with _ClassName__name, and we can access the variable using that name.

It allows us to access the class variables outside the class - in sub-classes, mostly used in inheritance.

Double Underscore in the Front and the Back¶

This convention is used for system-defined names that have a special meaning to the interpreter. For an example __init__ or __main__. Defining your own name may confuse the interpreter to cause unncecessary errors and should avoid doing that.

Sticking to the ones pre-defined by Python is certainly a good idea!

. . .¶

Other Naming Conventions¶

Packages: short, all-lowercase names, preferably no underscores. Eg: package

Modules: short, all-lowercase names, can have underscores. Eg: a_module amodule

Classes: CapWords - upper camel case convention. Eg: BankAccount

Functions: lowercase, words separated by underscores - snake_case Eg: open_account

Variables: lowercase, words separated by underscores - snake_case Eg: my_name, _name, _

Constants: all-uppercase, words separated by underscores Eg: NEW_CONSTANT
